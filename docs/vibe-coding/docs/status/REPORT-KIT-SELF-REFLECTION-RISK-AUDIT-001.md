## 1. System-Level Failure Modes

### FM-01: Step Closure Gate becomes boilerplate
- **Description:** Copilot fills the block mechanically without genuine evaluation. "Safe to proceed: YES" regardless of actual state.
- **Why:** Text template with no script enforcement. Copilot has strong pressure to move forward.
- **Likelihood:** HIGH
- **Impact:** HIGH — the gate's value is entirely in the judgment call. If it becomes checkbox theater, it actively misleads Stephen into thinking state was verified when it wasn't.

### FM-02: Startup Repair Sequence skipped via summary-and-stall
- **Description:** Copilot reads WARN output, proposes a plan, then asks "shall I proceed?" — the old behavior with one extra sentence prepended. Technically satisfies "propose one ordered repair sequence" while still pushing the decision.
- **Why:** The rule says "propose," not "execute."
- **Likelihood:** HIGH
- **Impact:** MED — friction reduced slightly but not eliminated.

### FM-03: Stale NEXT.md drifts undetected under WARN-only regime
- **Description:** Consumer repo's NEXT.md is 30+ days stale and describes completed work. Now WARN, not BLOCKED. Session proceeds. Copilot reads stale step as authoritative.
- **Why:** Contradiction detection is not scripted. Copilot may not compare NEXT.md content to git history if WARN is present.
- **Likelihood:** MED — particularly in slower-moving repos.
- **Impact:** HIGH — Copilot executes stale instructions, potentially redoing completed work or creating conflicting branches.

### FM-04: Generated vs. canonical misclassification
- **Description:** Copilot classifies a REPORT or research artifact as "Generated" and discards it at step boundary.
- **Why:** `docs/status/REPORT-*.md` files are in the canonical list by rule but may be generated by scripts in practice. Path-based classification is ambiguous at the boundary.
- **Likelihood:** LOW-MED
- **Impact:** HIGH — research artifacts lost.

### FM-05: Multiple BLOCKED gates — wrong one addressed first
- **Description:** Two or more gates BLOCKED; Copilot addresses the wrong one first, delaying repair.
- **Why:** Rule gives one example (`kit drift before packet staleness`) but no complete priority ordering.
- **Likelihood:** MED
- **Impact:** MED — repair takes longer; rarely causes data loss.

### FM-06: Step Closure Gate misapplied to protocol-maintenance prompts
- **Description:** protocol-v7.md exempts protocol-maintenance from some gate requirements. The Step Closure Gate has no such exemption. Copilot may apply it unnecessarily, OR skip it by incorrectly assuming the exemption applies.
- **Likelihood:** MED
- **Impact:** LOW-MED — primarily friction cost.

### FM-07: Evidence Pack pointer broken in consumer repos
- **Description:** `protocol-v7.md` now points to `standards/evidence-pack-template.md`. Consumer repos that copied protocol-v7.md but not `standards/` will have a dead pointer with no error.
- **Likelihood:** MED — depends on consumer subtree scope.
- **Impact:** MED — operators won't find the template; may reconstruct incorrectly or skip it.

### FM-08: check-prior-research.ps1 fix not propagated to consumers
- **Description:** Parser fix is in the kit. Consumers that haven't run `kit-update.ps1` still have the broken version. Prior research lookup silently fails.
- **Likelihood:** HIGH — every consumer repo that hasn't updated.
- **Impact:** HIGH — prior work lookup is a key gate; silent failure means duplicate work starts without checking.

### FM-09: Closure block "Required action: none" when tree is actually dirty
- **Description:** Copilot marks the tree clean from memory, not from `git status` output.
- **Why:** No script enforces the claim.
- **Likelihood:** MED
- **Impact:** HIGH — canonical changes lost at next step boundary.

### FM-10: Startup Repair re-run skipped
- **Description:** After fixing gates, Copilot skips step 4 (re-run session-start) and proceeds to feature work. Repair success is unconfirmed.
- **Likelihood:** MED-HIGH — most skippable step when repair "feels done."
- **Impact:** MED — missed failures from incomplete repair propagate into the session.

## 2. Human Factor Risks

### Does this actually reduce Stephen's cognitive load?

Partially. The Startup Repair Sequence eliminates the open-ended "what do you want to fix?" question — a real improvement. However:

- **Hidden burden shift:** Stephen must still ratify the proposed repair order. "Choose from a menu" becomes "ratify a plan" — lower burden but not zero.
- **Closure block volume:** 8 required fields × 3–4 steps per session = 24–32 fields to read per session. Volume drives skim behavior within 1–2 weeks.

### Where could Copilot still push decisions onto Stephen?

- **Startup Repair item 1:** "address the one blocking trust in repo state first" — Copilot may still ask "I believe kit drift should be addressed first, but do you want to confirm?" That is still a decision push.
- **Generated vs. canonical ambiguity:** Copilot may ask "this REPORT file — should I treat it as canonical?" Legitimate question, operator burden regardless.
- **Dirty tree at step boundary:** "Tree is dirty; should I commit or park?" — frequent, legitimate, but cumulative.

### Where might Stephen override or ignore the system?

- **In-flow overrides:** Stephen pastes a prompt and says "go" before the closure block is filled. Gate relies on Copilot self-enforcing; it won't always.
- **Partial repair acceptance:** If session-start reports 3 items, Stephen may say "just do X." The protocol has no language for partial repair acceptance; Copilot must decide whether this is a waiver or violation.
- **"I know it's fine" waiver:** "Only require update when actual mismatch is detected" (NEXT.md staleness rule) has no scripted trigger. Stephen can verbally waive it.

## 3. Workflow Edge Cases

### EC-01: Interrupted session mid-step
- **Expected:** Closure block incomplete; Safe to proceed: NO; Copilot returns closure actions only.
- **Risk:** A fresh session resets the condition. If context is lost, there is no closure block to evaluate. Work state must be reconstructed from `git log` manually. No recovery path is defined.

### EC-02: Dirty tree with generated + canonical changes
- **Expected:** Classification section distinguishes them; canonical committed/parked; generated may be discarded.
- **Risk:** `git diff` doesn't know which files are "generated." Copilot classifies from file path. `docs/status/REPORT-*.md` created mid-step by a script is at the boundary — canonical by rule, generated by origin.

### EC-03: Multiple BLOCKED gates
- **Expected:** Copilot addresses trust-in-repo-state gate first.
- **Risk:** Priority guidance gives one example but no complete ordered list. With three BLOCKED gates (e.g., kit drift + PAUSE.md expired + NEXT.md contradicts repo), ordering is genuinely ambiguous.

### EC-04: NEXT.md not updated for months
- **Expected:** WARN shown; Copilot reviews content and detects contradiction; issues BLOCK if mismatch found.
- **Risk:** Contradiction detection is not scripted. If the session starts in an unrelated feature lane, Copilot may not notice the contradiction. WARN is shown, work proceeds.

### EC-05: Incorrect DOCS_ROOT
- **Expected:** session-start.ps1 reads `$docsRoot` from config; all paths derive from it.
- **Risk:** If DOCS_ROOT is misconfigured, NEXT.md check targets wrong path, finds nothing, and skips silently. Under old BLOCKED regime the absence was salient. Under WARN-only, a skipped check produces no effective signal.

### EC-06: Consumer repo with intentional deviations
- **Expected:** Consumer-Kit Drift Gate reports drift; Startup Repair addresses it first.
- **Risk:** If a consumer has intentional rule overrides or local adaptations, the drift gate fires spuriously. Startup Repair Sequence says "fix drift first" — but fixing it may overwrite intentional customizations. No distinction between intentional and accidental drift exists in current tooling.

## 4. Tooling / Script Gaps

### TG-01: Step Closure Gate — no script enforcement
- Gate is a working-agreement rule only.
- **Risk:** Within 2–3 weeks, sessions under time pressure will skip the block. No audit surface to catch it.
- **Lightweight option:** Add `git log --format=%B -1` check in session-start.ps1 that warns if last commit message shows no evidence of closure block (soft signal only).

### TG-02: Startup Repair Sequence — not in session-start output
- The ordering rule exists only in the working agreement. session-start.ps1 does not print a prioritized repair list.
- **Risk:** Copilot may not recall the rule or apply it inconsistently.
- **Lightweight option:** Add a `Write-Host` block at the end of session-start.ps1 that prints a repair queue in order when BLOCKED/WARN items exist. Pure output, no logic change.

### TG-03: NEXT.md contradiction detection — not implemented
- BLOCK condition requires comparing NEXT.md content to git log. No script does this.
- **Risk:** The new WARN-only time rule depends on contradiction detection being trustworthy. A stale NEXT.md is now guaranteed WARN regardless of content mismatch.
- **Lightweight option:** Heuristic in session-start.ps1: if NEXT.md age > 21 days AND source files have commits newer than NEXT.md's last commit, emit an additional WARN: "NEXT.md has not been updated since code was committed — possible content mismatch."

### TG-04: Snapshot isolation — multi-script dependency
- Enforcement relies on `vibe-kit-snapshot.ps1`, `sync-forgpt.ps1`, and `session-start.ps1` all behaving correctly.
- **Risk:** If a consumer uses only a subset of these scripts, isolation may not apply. No single enforcement point.
- **Assessment:** Acceptable for now; new consumer onboarding should explicitly note the multi-script dependency.

## 5. Regression Risk vs v7.4.5

### What got safer
- **check-prior-research.ps1** works correctly — was silently failing before. Clear positive.
- **Step Closure Gate** creates an explicit dirty-tree check at every step boundary. No formal gate existed in v7.4.5.
- **Startup Repair Sequence** removes the open-ended choice question — observed real-world friction point.
- **Evidence Pack template** is findable by path rather than buried in a long protocol file.

### What got weaker
- **NEXT.md time-based BLOCKED is gone.** A 22-day stale NEXT.md would BLOCK in v7.4.5. Now it's WARN. Busy operators may dismiss WARN and proceed on stale state. This is the single most significant regression.
- **Step Closure Gate relies on Copilot self-enforcement.** In v7.4.5 there was no gate — but also no false assurance that one existed. Now there is a gate with no teeth, which can create false confidence.

### What became more ambiguous
- Generated vs. canonical file classification at step boundaries (new concept with no script grounding).
- "BLOCK if NEXT.md contradicts repo state" — the triggering condition is undefined operationally.
- "Multiple BLOCKED gates: address trust-blocking one first" — no complete priority ordering defined.

## 6. Top 3 Most Likely Future Headaches

### Headache 1: Consumer repo proceeds on stale NEXT.md step
- **Symptom:** Copilot starts implementation on a story completed 3 weeks ago. Work conflicts with already-merged code.
- **Root cause:** NEXT.md staleness changed to WARN-only. No contradiction detection. Busy session ignores WARN.
- **How it shows up:** Stephen notices the work doesn't match actual repo state: "wait, didn't we already do this?"

### Headache 2: Canonical research artifact discarded at step boundary
- **Symptom:** A REPORT or research doc classified as "Generated" in closure block. Copilot discards it. Lost on `git checkout .`.
- **Root cause:** Generated/canonical distinction is ambiguous for `docs/status/REPORT-*.md` files created mid-session by scripts.
- **How it shows up:** File is missing in the next session; work must be redone.

### Headache 3: Startup Repair re-run skipped; repair incomplete
- **Symptom:** Two gates BLOCKED, Copilot fixes both, skips step 4, proceeds to feature work. One gate was only partially fixed.
- **Root cause:** Step 4 (re-run) is the most rationalized-away step when repair feels complete.
- **How it shows up:** doc-audit.ps1 at end-of-session flags the issue; session must be partially rolled back.

## 7. Recommended Guardrails (Minimal)

### G-01: Add repair queue printer to session-start.ps1
When any gate returns WARN or BLOCKED, append a final block:
```
--- Suggested Repair Order ---
1. [BLOCKED] Consumer-Kit Drift Gate → run kit-update.ps1
2. [WARN] NEXT.md staleness (28d) → review before trusting
3. Sync forGPT packet last
4. Re-run session-start after repairs
```
**Effort:** ~15 lines of PowerShell output. No logic change. Makes the Startup Repair Sequence rule self-reinforcing.

### G-02: Tighten NEXT.md WARN message
- **Current:** "review before trusting; BLOCK only if content contradicts current repo state"
- **Problem:** "BLOCK only if content contradicts" is a judgment call with no guidance on how to check.
- **Fix:** Change to: "review Immediate Next Steps — if the listed step is not present in git history as incomplete work, BLOCK and update NEXT.md before proceeding"

### G-03: Add canonical/generated example table to Step Closure Gate
Add a two-line reference to the working-agreement clarification:
```
Canonical examples: REPORT-*.md in docs/status/, R-*.md in docs/research/, NEXT.md, protocol/*.md
Generated examples: docs/forGPT/*, VIBE-KIT-SNAPSHOT.md, any file output by sync-forgpt.ps1
```
**Effort:** 3 lines added to working-agreement-v1.md. Eliminates boundary ambiguity.

### G-04: Strengthen the re-run step in Startup Repair Sequence
- **Current:** "Re-run session-start — confirm all gates clear before proceeding."
- **Fix:** "Re-run session-start. This is the verification step. Do not propose feature work until the re-run returns no BLOCKED gates and all previously-failing WARN items are accounted for."
